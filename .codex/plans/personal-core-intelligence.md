## Goal and observable success

Complete the Personal Core direction with executable daily-use intelligence rather than prose-only claims. Rootloom must retrieve relevant project lessons, compute an explainable minimum risk tier from task/change signals, derive behavior-oriented verification recommendations, and include those results in the lightweight final summary without importing Enterprise Assurance machinery.

Success is observable when:

- `.project-memory/` keeps existing `rootloom-project-memory-v1` collections readable while explicit new records gain stable identity, evidence provenance, lifecycle state, expiry handling, and exact duplicate suppression;
- `project_memory.py context` can select relevant active failure, risk, and decision entries by path/query with bounded output and can surface stale-entry warnings without silently deleting or rewriting memory;
- a standard-library-only static analyzer reports changed paths, explainable signals, minimum recommended Tier/effective risk, matching project-memory warnings, and behavior-oriented verification recommendations;
- `finalize_change.py` uses the analyzer, prevents a manual risk declaration from lowering the detected floor, and adds the assessment and verification plan to the existing summary while retaining old command and summary fields;
- English/Chinese public docs, architecture, maturity claims, changelog, manifest, tests, and repository validation describe and enforce the same 2.1 contract;
- no Enterprise Assurance source, generated bytecode, product claim, setup surface, Hook, or CI job returns to the Personal Core implementation;
- focused tests, `make check`, compatibility smoke, diff review, and a fresh counterexample challenge pass.

## Non-goals

## Governed defect diagnosis

- Observed failure: advertised Task Intelligence, Engineering Memory, and Verification Intelligence are mostly instruction text; the executable path cannot retrieve relevant history, raise an underestimated risk declaration, or explain what behavior needs validation.
- Competing hypotheses: (1) stronger Skill prose alone is sufficient; (2) a small deterministic advisory layer is needed to make repeated daily decisions consistent. Current helpers and tests support (2): risk is a required free-form operator choice, context is all-or-nothing, and summary contains only executed commands.
- Ownership path: project-memory formats remain owned by `project-memory/scripts/project_memory.py`; change analysis and summary remain owned by `engineering-change/scripts/` and its small `runner/` helpers.
- Violated invariant: Personal Core claims must correspond to inspectable behavior that changes the user's next engineering decision.
- Root cause: 2.0 completed the enterprise split but left the highest-value personal intelligence as semantic workflow guidance rather than a bounded executable advisory contract.
- Root-cause alignment: PASS

## Constraints and invariants

- Runtime remains Python 3.11+ standard-library-only, local, deterministic for identical repository/memory state, bounded, network-free, and single-agent by default.
- Existing `rootloom-project-memory-v1` files remain readable; old entries without new metadata remain valid.
- Memory updates remain explicit, reviewable, atomic per file, and subordinate to current repository evidence. Context is read-only.
- Analyzer output is advisory and explainable. It may raise the effective floor but must not silently lower explicit risk or claim semantic certainty.
- Untracked file contents remain unread by finalization; analysis uses task text, paths, Git metadata, and tracked patch text only.
- Existing `--risk low|medium|high`, `diff.patch`, `test.log`, and summary fields remain accepted/readable. New fields are additive.
- Suggested verification is never automatically executed. Operator-supplied `--verify` commands remain the only executed commands.
- Enterprise Assurance remains owned by `codex/enterprise-assurance` and absent from `main` runtime/installation.

## Design and decisions

- Ownership: memory parsing/scoring/lifecycle stays in `project_memory.py`; task/change risk and verification recommendation stays in `engineering-change/scripts/runner/intelligence.py`; thin CLIs expose each owner.
- Interfaces: context accepts repeatable paths, query text, limit, and stale inclusion; records accept optional evidence/expiry and deterministic IDs; analyzer emits JSON with minimum Tier, effective risk, signals, memory matches, and verification plan; finalizer adds that payload and raises underestimated manual risk.
- Dependency direction: finalizer imports its local runner analyzer. The analyzer reads memory conservatively but never imports or mutates the project-memory writer, keeping Skills independently installable.
- Compatibility window: v1 memory files and legacy entries are accepted indefinitely for 2.x. New metadata is additive. Existing finalizer invocations continue working; `--risk` remains accepted, while omitted risk becomes automatic.
- Alternatives rejected: vector/database memory adds operational cost without proven personal value; Hook-time risk inference lacks task/diff context; path-only Tier rules overstate documentation/test changes; auto-running discovered commands creates execution authority and safety ambiguity; a second orchestration runner would recreate Assurance complexity.

## Rollout, failure, and rollback

- Dry-run/preview: `analyze_change.py` and `project_memory.py context` are read-only; memory writes remain explicit commands; finalization refuses output inside the captured repository and writes only to the requested external directory.
- Mixed-version behavior: old v1 collections and entries load unchanged. New entries remain JSON objects in the same v1 envelope; older code ignores unknown fields. Existing finalizer callers can keep declaring risk.
- Failure detection: strict collection/entry validation, expiry/date checks, deterministic tests, low-risk docs/test counterexamples, high-risk auth/migration/deletion cases, memory relevance/stale tests, and full repository contracts.
- Rollback or compensation: ordinary Git revert restores 2.0 behavior. Memory additions can be reverted as reviewable JSON; no automatic migration or destructive contraction occurs.
- Irreversible point: none. External publication is excluded pending separate authorization.

## Risks

- Risk: heuristic signals over-promote ordinary work.
  - Mitigation: report reasons, distinguish minimum Tier from semantic final judgment, treat docs/tests as context reducers, and cover counterexamples.
  - Residual risk: static analysis cannot understand all domain semantics; the active agent may raise the tier further.
- Risk: relevance scoring hides a useful lesson.
  - Mitigation: bounded default relevance plus explicit `--include-stale`, no deletion, transparent selection metadata, and no-query full active context.
  - Residual risk: terms and paths are lexical; users can inspect the complete files.
- Risk: additive v1 metadata surprises strict third-party consumers.
  - Mitigation: preserve envelope, existing fields, CLI, and semantic meanings; document additive compatibility.
  - Residual risk: undocumented exact-key consumers may require adjustment.
- Risk: verification recommendations appear stronger than executed evidence.
  - Mitigation: separate `verification_plan` from `tests` and `passed`; never auto-run suggestions.
  - Residual risk: a user may still treat a recommendation as proof unless they inspect `tests`.
